# Shikaku Race Studio 要件定義

## 1. 目的

本アプリは、`Square Race` 形式の観戦動画を短時間で量産するためのブラウザベース制作ツールである。  
ユーザーがこのアプリで行う主作業は「遊ぶ」ことではなく、以下である。

- コースを自動生成する
- 出力サイズを選ぶ
- レースを走らせる
- 画面を録画して動画素材として保存する

想定公開先:

- YouTube Shorts
- Instagram 正方形投稿

## 2. 対象ユーザー

- Shorts / Reels / SNS 向けに視聴用レース動画を量産したい制作者
- コース差分を大量に試したい運用者
- Seed ベースで再現可能なレースを管理したい運用者

## 3. 機能要件

### 3.1 出力

- 出力プリセットは以下の 2 種類に限定する
- `720 x 1280` : YouTube Shorts 用
- `720 x 720` : 正方形投稿用

### 3.2 レース

- レーサー数は `2` から `6`
- レーサーは正方形であること
- レーサーは初速を持って自動走行すること
- 全レーサーは同一の基準速度で走行すること
- レーサーは壁に衝突した際、反射して進行方向を変えること
- レーサー同士が接触した場合、壁衝突に準じる反発を互いに与えること
- 壁衝突時にはレーサーごとに異なる音が鳴ること
- 先頭の 1 台がゴールした時点でレース終了とする
- タイムアップは設けない

### 3.3 コース生成

- コースは `28 x 28` の固定グリッドで構成する
- スタート地点は `3 x 3` マスの正方形
- ゴール地点は `3 x 3` マスの正方形
- ゴール地点には白黒チェッカーマークを表示する
- コースは「障害物を並べる」のではなく「通路を掘る」方式で生成する
- 通路は必ずスタートからゴールまで連結していること
- 通路パラメータは以下とする
- `道の長さ`
- `幅の変化率`
- `Seed`
- `コーススタイル`

### 3.4 録画

- キャンバス映像を録画できること
- 可能な環境では MP4 で保存すること
- MP4 非対応環境では WebM へフォールバックすること

### 3.5 再現性

- Seed が同じであれば同一コースを再生成できること
- コース設定は JSON として出力・再読込できること

## 4. 非機能要件

- 動画上で見やすいことを最優先とする
- UI は制作補助に徹し、画面上の情報がコースを隠しすぎないこと
- 通路と壁の判別が一目でつく配色であること
- 生成結果は毎回大きく変わり、動画素材として差分が出ること

## 5. コース作成の詳細定義

### 5.1 基本構造

- コースは `28 x 28` セルの二値グリッドで表現する
- 各セルは `通路` または `壁` のどちらかである
- スタートとゴールは通路セルとして扱う
- レーサーが走行可能なのは通路セルの内部のみである

### 5.2 座標ルール

- 左上を `(0, 0)` とする
- 右方向に `x`、下方向に `y` が増加する
- スタートは左側に配置する
- ゴールは右側に配置する
- スタートとゴールは外周に接しすぎないよう `1` マス以上の余白を持つ

### 5.3 スタート / ゴール定義

- スタートは `3 x 3` の正方形セル領域
- ゴールは `3 x 3` の正方形セル領域
- スタート内部には複数レーサーを縦方向に分散配置する
- ゴールは視認性を優先し、白黒チェッカーで描画する

### 5.4 主経路生成

主経路は「右へ進みながら必要に応じて上下へ大きく迂回する」方式で生成する。

生成手順:

1. スタート中心点を決定する
2. ゴール中心点を決定する
3. スタート中心からゴール中心へ向かう中心線を作る
4. 中心線は以下の 2 種類の移動で構成する
5. `右へ進む水平移動`
6. `上または下へ迂回する垂直移動`
7. 最終的にゴール中心に必ず到達する

### 5.5 道の長さ

`道の長さ` は、中心線が何セル分移動するかの目標値である。

- 最低必要長は `スタートからゴールへの最短マンハッタン距離`
- 設定値が大きいほど、上下の迂回量が増える
- 設定値が小さいほど、より直線的な経路になる
- 実際の生成結果は目標値に近づけるが、完全一致は必須ではない

### 5.6 幅の変化率

`幅の変化率` は、中心線に沿って通路幅を変更する頻度を示す。

- `0%` に近いほど、通路幅はほぼ一定
- 値が高いほど、細い区間と太い区間が混在する
- 幅はスタイル定義ごとの最小幅と最大幅の範囲内で変化する
- 幅変化は中心線上の各区間単位で行う

### 5.7 コーススタイル

スタイルは、主経路の性格を決めるプリセットである。

- `snake` : 右進行の合間に大きな上下スイングを入れる
- `ladder` : 横移動は短く、縦移動が長い
- `canyon` : 幅広で比較的なだらか
- `zigzag` : 幅が細く、細かい折り返しが多い
- `variety` : 上記からランダム選択

各スタイルは以下を内部パラメータとして持つ。

- 水平移動の最大長
- 垂直スイングの最小長
- 垂直スイングの最大長
- 通路幅の最小値
- 通路幅の基準値
- 通路幅の最大値
- 縦方向へ振る傾向
- 交互に上下へ振るかどうか

### 5.8 通路の彫り方

- 中心線の各点に対して、現在の通路幅で正方形領域を掘る
- 隣接する中心線点の間は矩形接続して切れ目をなくす
- その結果、中心線は視覚的に連続した通路になる
- 掘られなかったセルは壁になる

### 5.9 妥当性条件

生成されたコースは以下を満たす必要がある。

- スタートからゴールまで通路が連結している
- 通路がグリッド外へはみ出さない
- 通路幅が 0 にならない
- スタート / ゴールが壁に埋まらない

## 6. レース進行の詳細定義

### 6.1 壁衝突

- レーサーが壁セルへ重なった場合、侵入前の方向へ押し戻す
- 衝突軸の速度を反転する
- 反射係数は 1 未満とし、完全なエネルギー保存にはしない

### 6.2 コース外脱出防止

- 毎フレーム、レーサーが通路セル上に残っているかを判定する
- 判定は中心点だけでなく、四隅と中心を含む複数サンプルで行う
- 通路外に出た場合は、最後に正常だった位置へ戻す
- 戻したうえで速度ベクトルを反射させ、停止しにくくする

### 6.3 レーサー同士の衝突

- レーサー同士が重なった場合、位置補正で分離する
- 相対速度が小さくても、壁衝突に準じる反射ベクトルを互いに与える
- 完全に重なった場合でも、決定的な法線ベクトルを作って分離する
- 分離後は基準速度へ正規化し、密着停止を防ぐ

## 7. タイムアップの定義

- タイムアップは設けない
- 旧実装の `28s` 固定終了は廃止する
- 現行仕様では、最初の 1 台がゴールした時点でレース終了とする

## 8. 既知の改善方針

- コース形状をさらに「枝道あり迷路型」へ寄せる余地がある
- 主経路の見栄えをもっと縦動画向けに最適化できる
- ゴール手前の演出や追い上げ演出は今後追加可能
